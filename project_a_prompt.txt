  DOCUMENT 4 — design.md  (ML Design Decisions)
=======================================================================

Write a complete ML design decisions document covering:

1. Why DistilBERT over BERT
   - 40% smaller, 60% faster, retains 97% of BERT performance
   - Fits within free-tier GPU memory (< 6GB VRAM)
   - Still captures contextual embeddings — far better than TF-IDF
   - Justify with benchmark references

2. Why GoEmotions dataset
   - 27 fine-grained emotions vs coarse pos/neg sentiment
   - Google-curated, Reddit domain, real informal text
   - Well-studied benchmark — easy to compare your results

3. Why CrossEntropyLoss with class weights
   - GoEmotions is heavily imbalanced (neutral >> rare emotions)
   - Class weights = total_samples / (num_classes * class_count)
   - Prevents model from always predicting neutral

4. Why macro F1 as primary metric
   - Macro F1 treats all 27 classes equally regardless of support
   - Forces the model to learn rare emotions, not just dominant ones
   - Standard metric for imbalanced multi-class classification

5. Why TF-IDF + LR as baseline (not zero-shot GPT)
   - Reproducible and deterministic
   - Widely understood by ML reviewers
   - Shows clear improvement curve: rule-based → statistical → neural
   - GPT baseline would obscure the fine-tuning contribution

6. Hyperparameter choices and justification
   - lr=2e-5: standard for BERT-family fine-tuning (Devlin et al.)
   - max_length=128: Reddit comments are short; >128 wastes compute
   - 4 epochs: diminishing returns beyond 3–4 for classification
   - Warmup 10%: prevents large gradient updates at start

7. Known limitations
   - Single-label prediction (GoEmotions is multi-label capable)
   - English only
   - Reddit domain bias (may underperform on formal text)

  DOCUMENT 8 — rules.md  (Project Rules & Constraints)

Write a complete rules and constraints document covering:

1. Reproducibility Rules
   - Always set torch.manual_seed(42), np.random.seed(42),
     random.seed(42), and transformers.set_seed(42) at top of
     every script
   - Never use random splits — use fixed stratified splits saved
     to disk, reload the same splits every run
   - Pin all dependency versions in requirements.txt
   - Save training args to training_args.json for auditability

2. Data Integrity Rules
   - Never use test set during training or hyperparameter selection
   - Baseline and DistilBERT must be evaluated on the exact same
     test split
   - Do not upsample or augment test data — only train data
   - Log dataset sizes (train/val/test) before every training run

3. Model Honesty Rules
   - Only report metrics on the held-out test set in the README
     and model card — not validation set numbers
   - Never cherry-pick the best epoch on test set —
     use best val F1 checkpoint evaluated once on test
   - If macro F1 < 0.45, do not claim "high accuracy" on resume —
     report the actual number

4. Code Quality Rules
   - Every function must have a one-line docstring
   - No hardcoded paths — use pathlib.Path and config dict
   - All results saved programmatically — no manual copy-paste
     of metric numbers
   - requirements.txt must be the only way to install dependencies

5. GitHub Rules
   - Commit message format: [phase] description
     e.g. [data] add GoEmotions loader with label mapping
          [train] add DistilBERT fine-tuning with Trainer API
          [eval] add confusion matrix and training curves
          [serve] add FastAPI inference endpoint
   - Never commit model weights to GitHub (>100MB) —
     add models/checkpoints/ to .gitignore
   - Do commit: all .py scripts, notebooks, results PNGs,
     metrics JSONs, README, model_card.md
   - README must have: project description, dataset, results table,
     how to run, sample output

6. Resume Rules
   - Only add this project to resume after Phase 7 is complete
   - The macro F1 number in the resume bullet must match
     results/metrics.json exactly
   - GitHub link in resume must point to the public repo
   - Do not list "DistilBERT" in skills — only list frameworks:
     PyTorch, HuggingFace Transformers

7. Time Management Rules (24-hour constraint)
   - If fine-tuning exceeds 5 hours, reduce to 2 epochs
     and document the constraint in model_card.md
   - Do not start serving (Phase 6) until evaluation (Phase 5)
     is fully complete
   - README and model_card.md must be finished before
     sharing the GitHub link anywhere
